# 선택과제 B - AWS 기초 이해

## 1. AWS 아키텍처 구성

현재 구현한 Simple Log Pipeline을 AWS 환경에서 운영한다고 가정했을 때, 아래와 같은 구조로 구성할 수 있다고 판단했습니다.

- **EC2**
    - Python 기반 로그 파이프라인 애플리케이션 실행
    - 이벤트 생성, DB 적재, SQL 분석, 시각화 수행

- **RDS for PostgreSQL**
    - 이벤트 데이터 저장
    - SQL 기반 분석 수행

- **S3**
    - 생성된 시각화 결과 이미지(PNG) 저장

- **CloudWatch**
    - 애플리케이션 로그 수집
    - 실행 상태 및 모니터링 확인

## 2. 서비스별 역할 설명

### EC2

애플리케이션 실행 환경으로 사용했습니다.  
현재 과제는 Python 스크립트 기반으로 동작하므로, 이를 가장 단순하게 운영할 수 있는 컴퓨팅 자원으로 EC2를 선택했습니다.

### RDS for PostgreSQL

이벤트 데이터를 안정적으로 저장하고 SQL 분석을 수행하기 위한 관리형 데이터베이스로 선택했습니다.  
로컬 PostgreSQL 환경을 AWS로 확장한다고 가정했을 때 가장 자연스럽게 연결할 수 있는 구조라고 판단했습니다.

### S3

시각화 결과물(PNG 파일)을 저장하는 스토리지로 사용했습니다.  
정적 파일 저장에 적합하고, 이후 웹 서비스나 리포트 시스템과 연계하기에도 유리합니다.

### CloudWatch

애플리케이션 실행 로그와 상태를 모니터링하기 위한 서비스로 선택했습니다.  
배치 실행 성공 여부 확인, 에러 로그 추적, 운영 관점의 기본 모니터링에 적합하다고 판단했습니다.

## 3. 가장 고민한 부분

가장 고민한 부분은 **애플리케이션 실행 환경을 EC2로 둘지, 더 관리형 서비스로 확장할지**였습니다.

이번 과제 범위에서는 구조를 단순하게 유지하고, 현재 구현한 Python 기반 로그 파이프라인과 자연스럽게 연결하는 것이 더 중요하다고 판단해 EC2를 선택했습니다.  
다만 실제 운영 환경에서는 실행 주기, 확장성, 운영 편의성을 고려해 ECS, Lambda, Step Functions 같은 더 관리형 서비스로 확장하는 방향도 가능하다고 생각했습니다.

## 4. 선택 이유 정리

이번 AWS 구성은 복잡한 인프라를 보여주기보다, 현재 구현한 로그 파이프라인을 **저장, 실행, 결과물 관리, 모니터링** 관점에서 AWS 서비스에 대응시킨 구조입니다.

즉,

- **EC2**: 애플리케이션 실행
- **RDS**: 데이터 저장 및 분석
- **S3**: 결과물 저장
- **CloudWatch**: 로그 및 모니터링

으로 역할을 나누어, 데이터 파이프라인 운영의 기본 구조를 설명할 수 있도록 구성했습니다.
